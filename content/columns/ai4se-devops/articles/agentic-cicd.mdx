## 发生了什么

CI/CD 流水线是 AI4SE 与 AI for DevOps 交汇最密集的场景。2026 年，Agent 技术开始从「辅助写代码」延伸到「管理交付过程」：

- **OpenAI Codex** 和 **GitHub Copilot Workspace** 将 Agent 能力嵌入代码审查与 PR 合并流程
- **Harness AI** 和 **GitLab Duo** 提供 AI 驱动的部署风险评估和回滚建议
- **Datadog Bits AI** 将可观测性数据反馈到 CI/CD 决策中，实现部署后的自动验证

这标志着 CI/CD 从「自动化」向「智能化」的演进：不仅是按脚本执行步骤，而是根据上下文做出判断。

## 为什么重要

传统 CI/CD 的核心假设是：**人类编写规则，机器执行规则。** 这套模式在稳定、可预测的环境中运行良好，但面对以下场景时力不从心：

1. **测试选择**：全量测试耗时过长，但如何智能选择「与本次变更相关的测试子集」？
2. **部署决策**：金丝雀发布中，哪些指标异常应该触发自动回滚？
3. **配置变更**：基础设施变更的影响范围难以完全预判
4. **安全审查**：依赖漏洞和配置风险的识别需要理解上下文

Agent 的核心价值在于 **在规则之外做判断**——理解变更意图、评估风险、在不确定性中做出合理决策。

## Agent 在 CI/CD 各环节的角色

### 代码审查阶段

AI Agent 不仅检查语法和风格，还能：

- 理解 PR 的业务意图，检查实现是否匹配
- 识别潜在的架构影响和安全风险
- 建议测试用例覆盖新增逻辑

这与 AI4SE 侧的能力直接重叠，是两条技术路线的自然汇合点。

### 构建与测试阶段

- **智能测试选择**：基于代码变更 diff 和历史失败模式，选择最相关的测试子集
- **自动生成测试**：为新增代码生成单元测试和集成测试草稿
- **Flaky test 检测**：识别不稳定测试并建议修复或隔离

### 部署阶段

- **风险评估**：综合代码变更、历史部署数据和当前系统状态，给出部署风险评分
- **渐进式发布控制**：根据实时指标自动调整金丝雀流量比例
- **自动回滚**：检测到异常指标时，在人工介入前执行回滚

### 部署后验证

- 关联部署事件与可观测性数据，自动判断部署是否成功
- 检测「静默失败」——服务运行但行为异常的情况
- 将部署后洞察反馈到下一次 CI/CD 决策

## 当前边界与风险

Agent 在 CI/CD 中的落地并非没有代价。需要清醒认识以下边界：

### 1. 判断错误的代价高昂

一个错误的自动回滚可能中断正常服务；一个被跳过的测试可能放过严重 bug。Agent 的决策需要 **可解释、可审计、可回滚**。

### 2. 上下文窗口的限制

CI/CD 决策需要理解整个系统状态——代码、配置、基础设施、历史事件。当前 Agent 的上下文能力仍不足以完全替代人类对复杂系统的全局判断。

### 3. 「自主」需要分级

并非所有 CI/CD 决策都适合交给 Agent。建议采用分级授权：

| 决策类型 | 建议授权级别 |
|---------|------------|
| 测试选择 | 可自主执行 |
| 代码风格修复 | 可自主执行 |
| 低风险配置变更 | 自主执行 + 事后审计 |
| 生产部署 | 建议执行 + 人工确认 |
| 生产回滚 | 可自主执行（安全优先） |

### 4. 与平台工程的依赖

如专栏第三期所述，Agent 在 CI/CD 中的可靠性直接取决于 IDP 的标准化程度和治理框架的完善度。没有 policy-as-code 约束的 Agent 自主部署，等同于没有安全带的自动驾驶。

## 落地建议

1. **从审查和测试入手**：这两个环节风险低、收益明显，适合作为 Agent 介入 CI/CD 的起点
2. **建立 Agent 决策的审计日志**：每次 AI 决策都应记录输入、推理和输出，支持事后复盘
3. **设置明确的自主边界**：用 policy-as-code 定义 Agent 可以自主执行的操作范围
4. **保持人类在环**：生产部署等高风险决策保留人工确认，但让 Agent 提供充分的决策依据

CI/CD 智能化不是一步到位的变革，而是在每个环节逐步引入 Agent 辅助决策、扩大自主范围的过程。关键是每一步都有数据支撑，每一个自主边界都有治理保障。
