## 为什么需要 AI 代码审查

人工 Code Review 有几个痛点：
- 审查者时间有限，容易遗漏细节
- 不同审查者关注的维度不同（有人关注安全，有人关注性能）
- 小团队没有专职审查者

AI 审查不能替代人工 Review，但可以作为**提交前的第一道防线**，帮你捕获明显问题。

## 审查工作流

```
你写完代码 → AI 预审查 → 修复 AI 发现的问题 → 提交 PR → 人工 Review
```

这个流程通常能帮你修复 30-50% 的明显问题，让人工 Review 聚焦在架构和业务逻辑上。

## 通用审查 Prompt

将以下 Prompt 保存为模板，每次提交前使用：

```
你是一位资深代码审查专家。请审查以下代码变更：

## 审查维度（按优先级排序）
1. **Bug 风险**：逻辑错误、边界条件、空值处理
2. **安全性**：注入攻击、敏感信息泄露、权限校验缺失
3. **性能**：不必要的循环、N+1 查询、内存泄漏
4. **可读性**：命名、函数长度、注释必要性
5. **最佳实践**：是否遵循项目的技术栈惯例

## 输出格式
按严重程度分类：
- 🔴 Critical（必须修复）
- 🟡 Warning（建议修复）
- 🟢 Suggestion（可选优化）

每条包含：位置、问题描述、修改建议

## 代码
[粘贴你的代码或 git diff]
```

## 在 Cursor 中使用

Cursor 内置了代码审查能力，操作更简单：

1. 选中改动的代码
2. `Ctrl+K` → 输入「审查这段代码的安全性和性能问题」
3. 或在 Agent 模式中：「审查我最近一次 commit 的变更，列出所有潜在问题」

## 分场景审查模板

### 安全审查（API 接口）

```
请从安全角度审查以下 API 接口代码：
- SQL/NoSQL 注入风险
- 认证和授权是否完备
- 输入验证是否充分
- 敏感数据是否加密/脱敏
- 错误信息是否泄露内部细节
```

### 性能审查（数据库查询）

```
请审查以下数据库相关代码的性能：
- 是否存在 N+1 查询
- 索引使用是否合理
- 大数据量下是否有分页
- 是否有不必要的全表扫描
```

## 注意事项

1. **AI 会误报**：不是所有 AI 指出的问题都是真问题，需要你有判断力
2. **不要审查超大变更**：单次审查建议不超过 300 行，大块变更拆分成多次
3. **结合自动化工具**：AI 审查 + ESLint/SonarQube 等静态分析工具互补

## 团队落地建议

- 在 PR 模板中加入「AI 审查完成」checkbox
- 将审查 Prompt 存入团队 Wiki，统一审查标准
- 定期回顾 AI 审查发现的问题，优化团队的编码规范
